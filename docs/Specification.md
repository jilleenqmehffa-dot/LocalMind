# AI 协作开发规范

本文档定义 LocalMind 项目中 AI 协作开发的工作边界、输出要求和质量标准。目标是在使用 AI 辅助开发时保持需求、设计、实现、测试和审查过程可追踪、可验证、可维护。

## 1. 基本原则

- 设计先行：在编写业务代码前完成需求、数据结构和接口契约设计。
- 小步迭代：每次变更聚焦一个明确目标，避免一次性生成大规模系统。
- 测试约束：关键功能必须先定义测试场景，再实现代码。
- 契约稳定：不得在未说明原因的情况下修改 API、数据库核心字段或模块边界。
- 错误透明：所有错误必须显式处理、记录日志，并返回统一格式。
- 安全优先：涉及认证、权限、输入处理、外部请求和模型调用的代码必须进行安全审查。

## 2. 标准开发流程

项目开发按以下顺序推进：

1. PRD：明确项目目标、用户角色、核心流程、MVP 范围和风险。
2. Schema：定义数据库 Schema、JSON Schema、类型、约束、索引和关系。
3. API：定义请求、响应、错误码、认证方式和限流策略。
4. Test：覆盖正常路径、边界条件、空值、异常和权限场景。
5. Code：在既定设计和测试约束下实现功能。
6. Review：检查可读性、边界条件、模块耦合和维护成本。
7. Security：审查注入、越权、敏感信息、Prompt Injection 等风险。
8. Deploy：补齐环境变量、启动方式、部署说明和运行验证。

## 3. 需求分析要求

开始实现前必须明确以下内容：

| 项目 | 说明 |
| --- | --- |
| 项目目标 | 本次功能要解决的问题和交付结果 |
| 用户角色 | 使用该功能的主体及其权限边界 |
| 核心功能 | 功能范围、输入输出和主流程 |
| MVP 范围 | 第一版必须完成的最小能力 |
| 数据流 | 数据从输入到存储、处理、输出的路径 |
| 流程图 | 复杂流程需提供 Mermaid 或等价结构图 |
| 风险分析 | 技术、安全、性能和交付风险 |

需求不明确或上下文冲突时，应暂停生成实现代码，先指出问题并补齐约束。

## 4. 数据建模要求

业务逻辑实现前必须完成数据建模。设计内容包括：

- 数据库表结构、字段类型、默认值和空值策略。
- 主键、外键、唯一约束、索引和级联策略。
- JSON Schema 或 Pydantic / TypeScript 类型定义。
- 数据生命周期：创建、更新、归档、删除和审计。
- 与现有表、服务和 API 的关系。

Schema 未确定前不得编写依赖该结构的业务逻辑。

## 5. API 设计要求

所有接口必须采用 OpenAPI / Swagger 风格描述，至少包含：

- Method 与 Path。
- Request Header、Path Parameter、Query Parameter 和 Body。
- Response Schema。
- Error Code 与错误响应格式。
- Auth 要求。
- Rate Limit 策略。

不得在接口契约未定义时直接生成 Controller 或业务处理函数。

## 6. 测试要求

测试用例应覆盖以下类型：

- 正常路径：合法输入和预期输出。
- 边界条件：最小值、最大值、空列表、长文本和特殊字符。
- 空值场景：缺失字段、空字符串、空文件和空响应。
- 异常路径：外部服务失败、数据库错误、网络超时和解析失败。
- 权限场景：未登录、无权限、跨租户访问和过期凭证。

代码合并前应确保相关测试通过。无法运行测试时，必须说明原因和剩余风险。

## 7. 代码质量标准

实现代码必须满足以下标准：

- 模块职责清晰，避免跨层调用和隐式依赖。
- 函数长度可控，复杂逻辑拆分为可测试的小单元。
- 命名准确，避免魔法数字、硬编码和重复逻辑。
- 类型定义完整，边界输入有校验。
- 错误处理明确，日志包含可定位问题的上下文。
- Python 遵循 PEP8，JavaScript / TypeScript 遵循项目既有规范。

单次生成的核心业务逻辑不应超过 50 行。复杂功能应拆分为多个可验证步骤。

## 8. 安全审查要求

生成或修改代码后必须检查以下风险：

| 风险类型 | 审查重点 |
| --- | --- |
| SQL 注入 | 查询参数化、ORM 使用方式、动态 SQL |
| XSS / CSRF | 前端输入输出、跨站请求、防护策略 |
| SSRF | 外部 URL 请求、内网地址访问、重定向 |
| Prompt Injection | 用户输入进入系统提示词或工具调用链的路径 |
| 越权访问 | 用户身份、资源归属、租户隔离 |
| Token 泄露 | 日志、错误响应、前端暴露、仓库提交 |
| 敏感信息泄露 | 文件、环境变量、数据库字段和调试输出 |

发现风险时必须标出位置、影响范围和建议处理方式。

## 9. 性能分析要求

涉及数据查询、批处理、模型调用、文件解析和向量检索的功能，应补充性能分析：

- 时间复杂度和空间复杂度。
- N+1 查询风险。
- 并发瓶颈和连接池配置。
- 缓存策略及失效机制。
- 大文件、长文本和高频请求的处理方案。

## 10. 文档要求

功能交付时应同步维护相关文档：

- README 或模块说明。
- API 文档。
- 环境变量说明。
- 启动方式。
- 部署说明。
- 测试和验证方式。

## 11. Git 规范

Commit 信息使用 Conventional Commits：

- `feat:` 新功能
- `fix:` 缺陷修复
- `docs:` 文档变更
- `refactor:` 重构
- `test:` 测试相关
- `chore:` 工程配置或维护任务

每次提交只处理一个独立主题，避免混合无关变更。

## 12. AI 输出格式

输出代码前必须先说明：

1. 设计原因。
2. 影响模块。
3. 主要风险。
4. 测试方案。

完整输出建议按以下顺序组织：

1. 分析
2. 方案
3. 风险
4. 测试方案
5. 代码
6. Review 建议

## 13. 禁止行为

AI 协作过程中禁止以下行为：

- 未经说明擅自升级依赖。
- 未经说明修改数据库核心字段。
- 未经说明修改 API 契约。
- 删除测试或日志。
- 隐藏错误或使用 silent fail。
- 一次性重构整个项目。
- 在上下文冲突时继续生成实现代码。

任何重大修改必须先说明原因、影响范围和回滚方式。
